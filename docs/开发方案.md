# CoTask 开发方案

> **项目主题**：CoTask —— AI 赋能的大学生课程小组协作辅助平台
> **技术栈**：Docker + Nginx + Vue 3 + Flask + MySQL（既定，不再调整）
> 本文档是项目的单一参考文档，覆盖产品定位、功能、页面/前端设计、系统架构、数据模型、关键流程、API 契约、AI 集成、测试、里程碑与部署。

---

## 1 项目概述

### 1.1 背景与痛点

大学课程小组作业普遍存在三大痛点：

1. **任务设置与拆解模糊**：组长难以把"课程要求"分解为可执行单元，组员不清楚自己做什么。
2. **进度失控**：群消息没人回、临近 DDL 才突击，缺乏可视的全局排期与预警。
3. **冷启动门槛高**：每次都从零开始，缺乏模板、案例、经验沉淀。

通用任务管理工具（Trello、Notion、飞书）无法贴合"高校 + 短周期 + 教育场景"的需求，市面专门工具（如"浙小组"小程序）功能扁平、缺 AI 能力、跨组视图弱。

### 1.2 产品愿景

CoTask 通过两个核心机制解决上述痛点：

- **管理愿景**：**AI 辅助的项目树** + **甘特图时间轴**，把学期任务递归拆解为有 DDL 的原子任务，实现透明、可追踪的协作。
- **灵感愿景**：**全链条渗透的灵感广场**，覆盖启动期（模板一键导入）→ 执行期（任务级参考案例）→ 总结期（沉淀分享），让集体智慧可复用。

### 1.3 用户与角色

| 角色                  | 描述                                                                 |
| --------------------- | -------------------------------------------------------------------- |
| **用户（User）**      | 所有注册的平台使用者                                                 |
| **组员（Member）**    | 加入某课程小组、执行被分配任务的成员，对本人任务有状态流转权         |
| **组长（Leader）**    | 小组管理者，拥有项目树/时间轴完全编辑权、成员管理权、解散权          |
| **内容贡献者/消费者** | 全体用户在灵感广场的泛化身份（浏览、上传、互动）                     |

> 角色是 **group-scoped**：同一用户在不同组可有不同角色，不存在"全局组长"。

## 2 与竞品的功能融合

竞品"浙小组"（小程序）的全部用户可感知功能均已并入 CoTask，结合本平台的 AI 能力做了升级：

| 竞品功能           | CoTask 中的对应实现                                                |
| ------------------ | ------------------------------------------------------------------ |
| 邀请码加入小组     | 组内邀请码 + 链接分享 + 二维码（H5 长按识别）                      |
| 任务分工可视化     | **项目树视图 + 成员泳道甘特图**，覆盖"分工 + 时间 + 状态"          |
| 多种任务分配方式   | 手动拖拽 / 随机分配 / **AI 按技能标签智能分配**                    |
| 进度同步           | 状态流转 + 父节点进度自动聚合 + WebSocket 实时推送                 |
| DDL 自动提醒       | Celery beat 定时扫 + 站内信 + 邮件 + **可选企业微信/钉钉 webhook** |
| 文件集中存放       | 任务节点级附件（MinIO），公共空间 + 小组私密空间                   |
| 讨论区             | **小组讨论区** + **任务级讨论串**，支持 @ / 引用 / 匿名            |
| 讨论信息通知       | 站内消息中心 + WebSocket 红点 + 邮件摘要                           |
| 戳一戳提醒（逾期） | "催办"按钮：发送站内消息 + 可选邮件，对方收到弹窗动画              |
| 模板数据库         | **灵感广场 → 项目树模板**（可一键导入）                            |
| 话术库             | **灵感广场 → 话术/脚本类目**（演讲稿、汇报模板）                   |
| 实用网址库         | **灵感广场 → 资源/工具类目**（带标签的链接收藏）                   |
| 自主上传资源       | 灵感广场发帖 + 任务附件 + 文件库                                   |
| 私密小组/公共空间  | 文件可见性：`private(group)` / `public(plaza)`                     |
| 信誉积分           | **贡献分系统**：按时完成 +、逾期 -、被点赞 +、被采纳 +             |
| 深色模式           | Element Plus 暗色主题 + CSS 变量 + 系统跟随                        |
| 一键登录           | 手机号短信登录 + 微信扫码登录（开放平台 OAuth）                    |
| 删除与撤销         | 软删除 + 30 秒 toast 撤销 + 操作历史回滚                           |
| 微信消息推送       | 通过订阅消息或企业微信机器人 webhook（v1.1）                       |

**CoTask 相对竞品的差异化**：

- **递归项目树**而非扁平任务列表，支持"先粗后细"的真实拆解过程。
- **跨组 Dashboard**：用户视角而非小组视角，统一展示所有课程的待办与 DDL。
- **AI 全链路**：文档→树、对话编辑、智能分工、每日建议、内容推荐。
- **任务级灵感关联**：执行某任务时直接看到相关参考案例，而不仅仅是独立的"模板中心"。

## 5 页面与前端设计

### 5.1 设计语言

- **UI 库**：Element Plus（深色主题成熟、表单/表格组件全面），自定义 CSS 变量层覆盖品牌色。
- **品牌色**：主色 `#3D7EFF`（理性蓝），辅色 `#FFB23D`（提醒橙），状态色 `success/warn/danger/info`。
- **字体**：默认 `-apple-system, "PingFang SC", "Microsoft YaHei", system-ui`；代码 `JetBrains Mono`。
- **栅格**：12 列，1240px 最大内容宽度，<768px 切换为单列。
- **图标**：`@element-plus/icons-vue` + 部分自定义 SVG（项目树节点状态）。
- **主题切换**：CSS 变量 + `prefers-color-scheme`，store 持久化到 localStorage。
- **国际化**：`vue-i18n`，默认 zh-CN，结构留好 en。

### 5.2 全局布局

```
┌────────────────────────────────────────────────────────────┐

├─────┬──────────────────────────────────────────────────────┤
```

未登录态：只渲染中央卡片，无侧栏。移动端：侧栏抽屉化，顶栏汉堡菜单。

### 5.3 页面详细设计

#### 5.3.1 登录/注册（/login）

- 居中卡片，Tab 切换"登录 / 注册"。
- 登录：账号（手机号或学号）+ 密码，"忘记密码"链接，"微信扫码登录"按钮。
- 注册：手机号 → 验证码 → 密码 → 姓名/学号/专业 → 完成。
- 错误提示在字段下方红字 + 顶部 toast。
- 失败 5 次后强制图形验证码（非锁定）。

### 5.5 状态管理（Pinia）

```ts
useAuthStore       // user, accessToken, refreshToken
useGroupStore      // currentGroupId, joinedGroups, currentRole
useTreeStore       // tree nodes (normalized), version, selectionId
useTimelineStore   // 派生自 treeStore，只保留叶子 + 视图窗口
useDashboardStore  // 跨组待办、AI 建议、组长报告
useInspirationStore// 列表、当前帖、筛选条件
useFilesStore      // 当前小组文件
useChatStore       // 讨论区 / AI 会话
useNotifyStore     // 通知列表、未读数
useUIStore         // 主题、侧栏折叠、抽屉栈
useWSStore         // 单例 WS 连接，事件总线
```

`TreeStore` 是核心，所有修改走 actions，actions 内做 **乐观更新 + 后端 PUT + 失败回滚**。WS 推送进来的 patch 直接 apply 到同一 store。

### 5.6 路由守卫与权限

- 全局前置：未登录跳 `/login`，登录态请求带 access token；401 自动刷新一次再失败才跳登录。
- `/groups/:gid/*` 守卫：拉取当前组成员关系，写入 `useGroupStore.currentRole`；非成员 403。
- 组件级权限：`v-if="role === 'leader'"` 或自定义 directive `v-leader`。

### 5.7 响应式与可访问性

- 三档断点：`<768px` 移动 / `768-1280` 平板 / `>1280` 桌面。
- 移动端：侧栏抽屉化、甘特图横向滚动、AI 助手底部弹起。
- 键盘可达：Tab 顺序合理、抽屉 Esc 关闭、树节点方向键导航。
- WCAG AA 对比度。

```
│   ├── modules/
│   │   ├── auth/              # 注册/登录/JWT/微信OAuth
│   │   ├── users/             # 资料、技能、贡献分
│   │   ├── groups/            # 小组 + 成员 + 邀请码 + 角色
│   │   ├── tree/              # 项目树 CRUD + closure 维护
│   │   ├── tasks/             # 原子任务状态、依赖、催办
│   │   ├── timeline/          # 甘特图聚合查询
│   │   ├── dashboard/         # 跨组聚合
│   │   ├── inspiration/       # 帖子、模板、互动
│   │   ├── files/             # MinIO 预签名 + 元数据
│   │   ├── discussion/        # 小组/任务讨论
│   │   ├── ai/                # LLM 适配 + prompt + schema
│   │   ├── notifications/     # 站内、邮件、WS 推送、催办
│   │   └── audit/             # 操作日志、回滚
│   ├── common/
│   │   ├── permissions.py     # @require_group_role
│   │   ├── errors.py
│   │   ├── pagination.py
│   │   ├── tx.py
│   │   └── soft_delete.py
│   └── tasks/                 # Celery tasks 定义
└── migrations/                # Alembic
```

设计原则：

- **service / repository 分层**：service 编排业务，repository 只读写；事务由 service 开。
- **依赖注入**：service 接收 repo/llm/ws 实例，便于单测 mock。
- **统一异常**：`AppError(code, http, message)`，全局 handler → JSON。
- **权限**：唯一装饰器 `@require_group_role('leader'|'member')` 从 path 取 `group_id`。
- **审计**：所有写操作经 `audit.record(actor, action, target, payload)`；关键操作可回滚。

#### tasks（项目树）

```sql
CREATE TABLE tasks (
  id           BIGINT PRIMARY KEY AUTO_INCREMENT,
  group_id     BIGINT NOT NULL,
  parent_id    BIGINT NULL,
  path         VARCHAR(512) NOT NULL,        -- 物化路径 '/1/4/9/'
  depth        SMALLINT NOT NULL,
  position     INT NOT NULL DEFAULT 0,
  title        VARCHAR(256) NOT NULL,
  description  TEXT,
  is_leaf      BOOLEAN NOT NULL DEFAULT FALSE,
  refined      BOOLEAN NOT NULL DEFAULT FALSE,
  start_date   DATE NULL,
  end_date     DATE NULL,
  status       ENUM('todo','in_progress','done','blocked')
               NOT NULL DEFAULT 'todo',
  progress     TINYINT NOT NULL DEFAULT 0,
  version      INT NOT NULL DEFAULT 1,       -- 乐观锁
  created_at   DATETIME DEFAULT CURRENT_TIMESTAMP,
  updated_at   DATETIME ON UPDATE CURRENT_TIMESTAMP,
  deleted_at   DATETIME NULL,
  INDEX idx_group_parent (group_id, parent_id),
  INDEX idx_path (group_id, path),
  INDEX idx_ddl (group_id, end_date),
  CONSTRAINT chk_leaf_dates CHECK (
    is_leaf = FALSE OR (start_date IS NOT NULL AND end_date IS NOT NULL)
  )
);

CREATE TABLE task_closure (
  ancestor_id   BIGINT NOT NULL,
  descendant_id BIGINT NOT NULL,
  distance      SMALLINT NOT NULL,
  PRIMARY KEY (ancestor_id, descendant_id),
  INDEX idx_desc (descendant_id)
);

CREATE TABLE task_assignments (
  task_id BIGINT NOT NULL,
  user_id BIGINT NOT NULL,
  PRIMARY KEY (task_id, user_id),
  INDEX idx_user (user_id)
);

CREATE TABLE task_dependencies (
  task_id    BIGINT NOT NULL,         -- 后继
  depends_on BIGINT NOT NULL,         -- 前驱
  PRIMARY KEY (task_id, depends_on)
);

CREATE TABLE task_inspiration_refs (
  task_id BIGINT NOT NULL,
  post_id BIGINT NOT NULL,
  source  ENUM('ai','manual') NOT NULL DEFAULT 'manual',
  PRIMARY KEY (task_id, post_id)
);
```

**三种树查询模式各取所长：**

| 查询                       | 用什么                          |
| -------------------------- | ------------------------------- |
| 渲染整树                   | 按 `path` 排序一次取出           |
| 某节点的所有子孙           | `task_closure` JOIN             |
| 父子增删改                 | `parent_id`，事务同步 closure   |
| "我有关的节点"含祖先链     | `task_assignments` JOIN closure |

## 9 关键流程

### 9.1 项目树 AI 生成（异步 + 流式）

```
组长 → POST /api/ai/jobs  {scope:'tree_gen', input:{doc_id|text}}
         ← 202 {job_id}
       (Worker 拉文档 → 调 LLM)
组长 ← WS push  {job_id, status:'streaming', delta:'...'}
组长 ← WS push  {job_id, status:'done', result:{tree_json}}

组长 → POST /api/groups/:gid/tree/from-job  {job_id, tree_json}
         ← 201 {tree_root_id}
```

LLM 返回严格 JSON Schema（Pydantic 校验）。Worker 写库前：
1. Schema 校验
2. 深度 ≤ 5、节点 ≤ 200 的硬上限
3. 标题清洗、去空白、去重
4. 标签 `is_leaf` 推断（最深节点为叶）

### 9.2 项目树 AI 对话编辑（纯函数模型）

```
edit(current_tree, instruction) → (new_tree, diff)
```

服务端不维护对话状态，每次都传当前完整树。优势：撤销简单、跳页不丢、重试幂等。

UI：前端展示 diff（绿增 / 红删 / 黄改），用户确认后 `PUT /api/groups/:gid/tree`。服务端：
1. 比对 `version` 乐观锁；冲突返回 409 + 当前版本
2. 在事务里 apply diff，同步 `path / depth / closure`
3. 写 `audit_log`
4. WS 广播 `tree.updated` 给本组在线成员

### 9.3 状态推进 + 进度聚合 + 贡献分

组员推进任务：
```
PATCH /api/tasks/:id/status  {status:'done'}
  ↓
service:
  1. 鉴权（必须是 assignee 或组长）
  2. 写状态 + version+1
  3. 沿 closure 向上重算父节点 progress
  4. 若按时完成且 status:'done' → 贡献分 +N
  5. WS push task.status_changed
  6. audit_log
```

### 9.4 催办（戳一戳）

```
POST /api/tasks/:id/nudge
  ↓
检查 24h 内是否已催 → notifications + WS 弹窗 + 可选邮件
对方端：dashboard 震动图标 + 任务高亮
```

### 9.5 DDL 自动提醒（Celery beat）

- 每 10 分钟扫一次 `tasks WHERE is_leaf AND status != 'done' AND end_date - now BETWEEN 0 AND 72h`
- 命中任务 → 写 notifications + 推 WS + 视用户偏好推邮件
- 同一 (user, task) 24h 内最多 1 条，使用 Redis SETNX

### 9.6 文件上传

```
前端 → POST /api/files/sign  {filename, size, md5}
       ← {upload_url, file_id}   (MinIO 预签名 PUT，10min)
前端 → PUT upload_url (直传 MinIO)
前端 → POST /api/files/:id/finalize
       ← {url, version}
```

md5 命中已有文件 → 秒传，直接返回现有 storage_key。

### 9.7 模板导入

```
POST /api/inspiration/posts/:id/import  {to_group_id}
  ↓
service:
  1. 鉴权：目标组组长
  2. 若目标组已有树 → 询问"合并/替换/作为子节点"
  3. 深拷贝 template_root_id 子树 → 新 tasks（新 ID / 新 path）
  4. 同步 closure
  5. 清空 assignments / dates（导入者重新分配）
  6. audit_log
```

### 9.8 权限模型

- 唯一装饰器 `@require_group_role('leader'|'member')`
- 不在 JWT 里塞 role；每次请求查 `group_members`，按 `(uid, gid)` Redis 缓存 60s。
- 组长关键操作（解散、转让、批删）走二次确认 + `confirm_token`（2 分钟）。

### 9.9 认证

- JWT 双 token：access 15min（Authorization header），refresh 30d（HttpOnly cookie）
- 微信登录：OAuth 取 openid → 若已绑定取用户，否则要求绑定手机号建账号
- 限流：同账号+IP，5 分钟 10 次失败要求图形验证码（不锁定）

## 14 非功能与安全

- **配置**：secret → 环境变量 + docker secrets；`.env.example` 入库，`.env` 不入库。
- **限流**：Nginx 层 IP 限流 + 应用层基于 Redis 的接口级限流。
- **审计**：组长的"解散/转让/批删/树覆盖"写 `audit_log`，永久保留。
- **数据导出**：用户可导出本组项目树（JSON / Markdown），满足"数据可携带"。
- **日志**：JSON 结构化日志 → Loki / 文件；trace_id 贯穿前后端。
- **备份**：MySQL 每日 mysqldump + binlog；MinIO 开启版本化。
- **隐私**：匿名发言对组员匿名，对组长可逆向（避免恶意言论）；账号注销 30 天软删。

## 15 与竞品（"浙小组"）的定位对比

| 维度         | 浙小组（竞品）           | CoTask                                       |
| ------------ | ------------------------ | -------------------------------------------- |
| 任务模型     | 分工列表 + 文件夹整合    | **递归项目树 + 原子任务时间轴**              |

| 内容沉淀     | 模板/话术/网址库         | **同覆盖 + 任务级灵感推荐 + 一键导入项目树** |
| 协作信号     | 戳一戳、信誉积分         | **戳一戳 + 贡献分 + WS 实时 + AI 风险预警**  |
| 文件         | 私密小组 / 公共空间      | **同 + 任务级附件 + 版本 + 秒传**            |
| 讨论         | 讨论区（含匿名）         | **同 + 任务级讨论 + @ + 引用**               |
| DDL 提醒     | 微信提醒（小程序内）     | **站内 + 邮件 + 微信/企业微信（v1.1）**      |
